# MATLAB MAT-file reader

Reads MATLAB MAT-files (`.mat`, versions 5 through 7.2) in pure Power Query
M. No MATLAB licence, no Python interpreter, no COM automation.

Power BI has no native MAT-file connector. The usual answers are exporting
to CSV from a MATLAB licence someone else owns, or a Python script step with
scipy (which needs a configured interpreter and rules out most Service
refresh scenarios). Test benches, lab instruments, simulation runs and
university research pipelines all save results as `.mat`; this reader parses
the file format directly, so there is nothing to install.

## Usage

Paste [`Matlab.Document.pq`](Matlab.Document.pq) into a blank query and name
the query `Matlab.Document`. Then:

```m
let
    Source = File.Contents("C:\data\results.mat"),
    Doc    = Matlab.Document(Source),
    Data   = Doc{[Name = "measurements"]}[Data]
in
    Data
```

The result is a navigation table with one row per variable, showing each
variable's name, class and dimensions. File-level metadata (header text,
byte order, subsystem data offset, variable count) is attached as table
metadata (`Matlab.HeaderText`, `Matlab.ByteOrder`, ...).

### How variables convert

MATLAB arrays are column-major and can hold anything from a scalar to a
nested struct of cell arrays. The conversion is by shape, applied
recursively:

| MATLAB value | Becomes |
|---|---|
| 1x1 array | the scalar itself (complex: a `[Re, Im]` record) |
| 1xN or Nx1 vector | a list (char row vector: a text) |
| MxN matrix | a table with columns `C1`..`CN`, one row per matrix row |
| N-dimensional array | a long table `D1`..`Dk`, `Value`, column-major order |
| char matrix | a list of M texts, one per row |
| 1x1 struct | a record of fields |
| struct array | a table, one row per element, one column per field |
| cell array | shaped like a numeric array, elements converted recursively |
| sparse matrix | a table `{Row, Column, Value}`, 1-based, CSC order |
| any empty array | null (empty char: `""`) |

Numeric columns are typed: `Int64.Type` for the integer classes, `number`
for single and double, `logical` for logicals. The class decides the type
even when the file stores the data in a narrower type (MATLAB's numeric
storage compression).

### Options

Second argument, optional record, all keys optional:

| Key | Default | Effect |
|---|---|---|
| `Encoding` | 1252 | `TextEncoding` or codepage number for char data stored in the legacy 8-bit form (miINT8/miUINT8). UTF-8/UTF-16/UTF-32 char data carries its own encoding. |
| `Strict` | `false` | Error on tolerated malformations: an unknown top-level element type, a compressed element that does not hold a matrix, a function handle. |

## Supported

- MAT-file level 5 files as written by MATLAB v5 through v7.2 (`save -v6`,
  `save -v7`) and by scipy.io.savemat.
- Both byte orders: "IM" little-endian (every file written this century) and
  "MI" big-endian (old Solaris/PPC MATLAB).
- Compressed variables (the v7 default: zlib per top-level element) and
  uncompressed (v6).
- All numeric classes int8 through uint64, single, double; logical; char in
  all five storage encodings (miUINT16, miUTF8, miUTF16, miUTF32, legacy
  8-bit); complex arrays; N-dimensional arrays; cell arrays; structs and
  struct arrays; objects (read as structs, class name reported); sparse
  matrices, including sparse logical and sparse complex.
- Numeric storage compression (a double array stored as int8 on disk).

## Not supported

- **v7.3 files.** `save -v7.3` writes an HDF5 container, a different format
  entirely. The reader detects it and says so; re-save with
  `save('file.mat','-v7')`.
- **v4 files.** Pre-1996 layout, detected with a clear error. Re-save with
  any modern MATLAB.
- **Function handles and opaque/Java objects.** No data representation;
  they load as null (error under `Strict`). The subsystem data blob MATLAB
  appends for them is skipped.
- **int64/uint64 values beyond 2^53.** M numbers are doubles; larger
  integers lose precision.

## How it works

A MAT v5 file is a 128-byte header followed by data elements, each an
8-byte tag (type, byte count) plus payload. A variable is a `miMATRIX`
element whose subelements hold the array flags, dimensions, name and data;
container classes (cell, struct) nest further `miMATRIX` elements inside.
In v7 files each top-level element is wrapped in `miCOMPRESSED`: a zlib
stream, which is a 2-byte header plus a raw deflate body plus an Adler-32
trailer. The reader strips the 2 bytes, inflates with
`Binary.Decompress(..., Compression.Deflate)` and ignores the trailer.

Everything else is `BinaryFormat.*` reads over a buffered binary: the tag
walk, the small-element format (type and length packed into 4 bytes),
column-major reshaping, and the sparse ir/jc/pr column pointers.

## Testing

Fixtures live in [`test/`](test/), generated by
[`test/make_fixtures.py`](test/make_fixtures.py) with scipy plus four
hand-crafted files scipy cannot write (big-endian, miUINT16 char, numeric
storage compression, header-only). [`test/expected.md`](test/expected.md)
describes what each fixture proves. The parse logic is mirrored in
[`test/mirror.py`](test/mirror.py) and cross-validated cell-by-cell against
`scipy.io.loadmat` by [`test/check_mirror.py`](test/check_mirror.py).
