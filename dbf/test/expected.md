# DBF fixtures: what each file proves

All fixtures are synthetic, generated by `make_fixtures.py` in this directory.
Regenerate with:

```
python3 -m venv venv && venv/bin/pip install dbf dbfread
venv/bin/python make_fixtures.py
```

Most are written by the `dbf` package (Ethan Furman's pure-Python xBase
writer); two are hand-crafted byte by byte because no open writer produces
them. The parse logic destined for `Dbf.Table.pq` is mirrored in `mirror.py`
and validated by `check_mirror.py` on every fixture, both cell-by-cell against
dbfread (an independent reference reader) and against hand-written expected
literals.

To test, load a fixture through `Dbf.Table`:

```m
Dbf.Table(File.Contents("...\dbf\test\memo3.dbf"),
          [Memo = File.Contents("...\dbf\test\memo3.dbt")])
```

## people.dbf: dBASE III (0x03), 3 active rows + 1 deleted, 5 fields

| NAME (C12) | QTY (N6.0) | PRICE (N8.2) | BORN (D) | OK (L) |
|---|---|---|---|---|
| Ada Lovelace | 42 | 19.99 | 1815-12-10 | true |
| trim  me | -7 | -0.5 | 2026-07-17 | false |
| (empty) | null | null | null | null |

A fourth row (`deleted row`, 1, 1.0, 2000-01-01, true) is flagged deleted: it
must be absent by default and present with a true `_Deleted` column under
`[IncludeDeleted = true]`. Proves: trailing-space trim that keeps interior
spaces (`trim  me`), blank numerics/dates/logicals as null, integer typing for
N with no decimals, and the codepage fallback (the language-driver byte is
zeroed; text must still decode as 1252).

## memo3.dbf + memo3.dbt: dBASE III memo (0x83), 4 rows

| TITLE | NOTE |
|---|---|
| short | short memo |
| long | `0000-abcdef 0001-abcdef ...` (1200 chars, spans three 512-byte blocks) |
| empty | (empty string: pointer to a zero-length entry) |
| blank | null (pointer field is all spaces) |

Proves: 10-character right-justified block pointers, 0x1A-terminated entries,
memos spanning blocks, and the difference between an empty memo and no memo.
Without `options[Memo]` the NOTE column must be all nulls, not an error.

## fox.dbf + fox.fpt: FoxPro 2.x (0xF5), 2 rows

| NAME | NOTE (M) | PIC (P) |
|---|---|---|
| one | text memo | 13 bytes starting `89 50 4E 47` (binary) |
| two | (empty) | null |

Proves: the .fpt layout (big-endian block size at offset 6, per-entry type and
length), type 1 entries decoding to text, type 0 entries staying binary, and a
blank picture pointer as null.

## vfp.dbf + vfp.fpt: Visual FoxPro (0x30), 3 rows, 8 visible fields

| NUM (I) | DBL (B) | CUR (Y) | WHEN (T) | NAME (C, null) | CNT (I, null) | OK (L) | NOTE (M) |
|---|---|---|---|---|---|---|---|
| 7 | 2.5 | 12.3456 | 2026-07-17 13:45:30 | alpha | 42 | true | vfp memo |
| -2000000000 | -0.125 | -99.99 | 1970-01-01 00:00:00 | null | null | false | (empty) |
| 2000000000 | 10000000000 | 0 | null | (empty) | 0 | null | `v` x 500 |

Proves: int32, double, currency (int64 / 10000), datetime (Julian day +
milliseconds since midnight, all-zero bytes as null), 4-byte memo pointers,
and the `_NullFlags` hidden field: it must not appear as a column, row 2's
nulls come from its bits, and row 3 shows empty-but-not-null values for the
same fields. The 263-byte VFP header backlink must not upset field parsing.

## empty.dbf: dBASE III, 0 rows

A dictionary with no records. Must yield an empty table that still has both
typed columns (`A` text, `B` whole number), not an error.

## memo4.dbf + memo4.dbt: hand-crafted dBASE IV (0x8B), 2 rows

| TITLE | NOTE (M) | WIDE (C300) | AMT (N8.2) |
|---|---|---|---|
| alpha | dbase four memo | `W` x 280 | 123.45 |
| beta | null | (empty) | -99.99 |

Hand-built because no open writer emits the dBASE IV memo layout. Proves: the
FF FF 08 00 entry signature with a little-endian length that includes the
8-byte entry header, a Char field wider than 255 bytes (length is stored as
length + 256 x decimal_count), and a comma decimal separator in a numeric
(`123,45` in the file). dbfread reads this file too, so it is cross-checked.

## varchar.dbf: hand-crafted Visual FoxPro 0x32, 3 rows, 9 visible fields

| VC (V10) | VCN (V8, null) | VB (Q6) | C1 | C2 | C3 | C4 | C5 | NUM |
|---|---|---|---|---|---|---|---|---|
| ABCDEFGHIJ | xy | 01..06 (6 bytes) | aa | null | cc | (empty) | ee | 1 |
| AB | null | 01 02 | null | bb | null | dd | null | -5 |
| (empty) | FULLFULL | (0 bytes) | zz | (empty) | (empty) | (empty) | (empty) | 0 |

Hand-built because no open writer supports Varchar/Varbinary. Bit layout per
the VFP documentation: each V/Q field owns a varlength bit (0 = full, 1 =
actual length in the field's last byte); a nullable V/Q owns a second, higher
null bit; plain nullable fields own one bit each. Nine bits total, so
`_NullFlags` is two bytes and row 2's C5 null sits in the second byte. Proves:
full/partial/empty varchar and varbinary, varchar values that must not be
space-trimmed, null bits across both bytes, and that the trailing `NUM` column
still decodes (offsets survive). dbfread cannot arbitrate here (it rejects
type Q); the expected literals in `check_mirror.py` are the reference.
