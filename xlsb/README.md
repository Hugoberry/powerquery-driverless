# Excel Binary Workbook (.xlsb) reader

Reads Excel Binary Workbooks (`.xlsb`, BIFF12) in pure Power Query M. No ACE
OLEDB provider, no gateway software, no custom connector.

Power Query reads `.xlsb` through the Access Database Engine (ACE). That works
on a Desktop machine that has ACE, but ACE cannot be installed in cloud
environments — so in Power Query Online a `.xlsb` file forces a gateway even
when the file already sits in SharePoint or Blob Storage, somewhere the Service
could reach directly. This reader parses the binary format itself, so the same
file refreshes with nothing installed and no gateway in the middle.

Correctness, not just portability: ACE infers each column's type by sniffing
the first rows and returns nulls for values that don't match its guess — the
classic mixed-column data loss. This reader decodes every cell from its record
type and never drops a value.

## Usage

Paste [`Xlsb.Workbook.pq`](Xlsb.Workbook.pq) into a blank query and name the
query `Xlsb.Workbook`. Then:

```m
let
    Source = File.Contents("C:\data\report.xlsb"),
    Wb     = Xlsb.Workbook(Source),          // navigation table of sheets
    Sheet1 = Wb{[Name = "Sheet1"]}[Data]
in
    Sheet1
```

The result is a navigation table with one row per worksheet (`Name`, `Data`,
`Hidden`, plus navigator plumbing). Each `Data` cell is the sheet as a table
with columns `Column1..ColumnN`, like `Excel.Workbook` without header
promotion. `Xlsb.Date1904` in the table metadata reports the date system.

### Options

Second argument, optional record, all keys optional:

| Key | Default | Effect |
|---|---|---|
| `IncludeHiddenSheets` | `false` | List hidden and very-hidden sheets (the `Hidden` column marks them) |
| `PromoteHeaders` | `false` | Use each sheet's first row as column headers |
| `DetectDates` | `true` | Convert date/time-formatted numbers to `datetime`; `false` returns raw serials |
| `MaxRows` | all rows | Return at most N rows per sheet |
| `Strict` | `false` | Error on tolerated malformations (out-of-range shared-string index) |

## Supported

- All BIFF12 cell records: RK-packed and full-precision numbers, shared and
  inline strings, booleans, error cells, styled blanks, and cached formula
  results (number, string, boolean, error).
- Date/time detection from builtin format ids and custom format codes,
  including the 1904 date system.
- Hidden and very-hidden sheets; chart/macro/dialog sheets are recognised and
  skipped.
- Stored and deflated ZIP entries, parsed from the central directory.

## Limitations

- **Encrypted workbooks** are CFB containers rather than ZIP archives and
  produce a clear error.
- **Formulas are not evaluated**; the cached result Excel stored is returned.
- **Error cells** surface as cell-level errors (like `Excel.Workbook`), with
  the Excel error name (`#DIV/0!`, `#N/A`, ...) in the message. Use
  `Table.RemoveRowsWithErrors` or `try` to handle them.
- **Excel's leap-year bug:** serials 1..59 are mapped so dates display as
  Excel shows them; the fictitious 1900-02-29 (serial 60) becomes 1900-03-01.
- **Memory:** the whole file is buffered; peak memory is a multiple of file
  size.
- Pivot caches, tables, comments, shapes and other non-cell content are
  ignored. Zip64 archives error clearly (real `.xlsb` files are not Zip64).

## How it works

An `.xlsb` file is an OPC/ZIP package whose parts are binary BIFF12 record
streams instead of XML. The reader locates the end-of-central-directory
record, walks the central directory, and inflates parts with
`Binary.Decompress(..., Compression.Deflate)` (ZIP entries are raw RFC 1951
streams). `xl/workbook.bin` yields the sheet list and 1904 flag; sheet parts
are resolved through `xl/_rels/workbook.bin.rels` (the one XML part touched,
read with `Xml.Document`). Each BIFF12 record is a variable-length id plus a
7-bit varint length; cell records share a header of column and style index,
and a `BrtRowHdr` record carries the current row. Style indexes map through
`xl/styles.bin` cell XFs to number formats for date detection.
