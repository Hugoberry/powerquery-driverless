# MBTiles fixtures: what each file proves

All fixtures are synthetic, generated by `make_fixtures.py` in this
directory with the Python standard library only (sqlite3 + zlib + struct).
Regenerate with:

```
python3 make_fixtures.py
```

To test, paste `sqlite3/Sqlite3.Database.pq` as a query named
`Sqlite3.Database` (the reader depends on it), then load a fixture through
`Mbtiles.Document`:

```m
Mbtiles.Document(File.Contents("...\mbtiles\test\raster.mbtiles"))
```

Every fixture must produce a navigation table with exactly two entries,
`Metadata` and `Tiles` (none of them has UTFGrid tables).

## raster.mbtiles: plain layout, format png

Metadata: name `raster-demo`, format `png`, bounds, center, minzoom 0,
maxzoom 1, type `baselayer`, version `1.1` (8 rows).

Tiles (TMS rows as stored, `tile_row_xyz` computed):

| zoom_level | tile_column | tile_row | tile_row_xyz | tile_data |
|---|---|---|---|---|
| 0 | 0 | 0 | 0 | 1x1 red PNG |
| 1 | 0 | 1 | 0 | 1x1 green PNG |
| 1 | 1 | 0 | 1 | 1x1 blue PNG |

The three PNGs are valid, distinct, and all start `89 50 4E 47`; none is
gzip, so auto-decompression must leave every blob untouched. Proves: the
plain `tiles` table path, the TMS-to-XYZ row conversion at two zoom levels,
and that raster blobs pass through byte-identical.

## vector.mbtiles: format pbf, gzip auto-detection

Metadata: name `vector-demo`, format `pbf`, minzoom 0, maxzoom 1, and a
`json` key with a vector_layers stanza (5 rows).

| zoom_level | tile_column | tile_row | tile_row_xyz | tile_data |
|---|---|---|---|---|
| 0 | 0 | 0 | 0 | 18 bytes, exactly `fake-mvt-payload-A` as UTF-8 |
| 1 | 0 | 0 | 1 | 15 bytes, exactly `raw-not-gzipped` as UTF-8 |

Row 1 is stored gzipped (as the MBTiles spec requires for pbf) and must come
back decompressed by default; row 2 is stored raw and must pass through
untouched, proving the per-tile signature sniff rather than a blanket
format=pbf rule. Under `[DecompressTiles = false]` row 1 must come back as
the stored 38-byte gzip stream (starts `1F 8B`). Under
`[DecompressTiles = true]` row 2 must error.

## deduped.mbtiles: tippecanoe layout (map + images, tiles is a view)

Metadata: name `deduped-demo`, format `png`, minzoom 1, maxzoom 1 (4 rows).

There is no `tiles` table, only a `tiles` view over `map` (zoom_level,
tile_column, tile_row, tile_id) and `images` (tile_id, tile_data). The
reader must detect the layout and perform the join itself (compare as a set;
join row order is not significant):

| zoom_level | tile_column | tile_row | tile_row_xyz | tile_data |
|---|---|---|---|---|
| 1 | 0 | 0 | 1 | 1x1 blue PNG (`sea`) |
| 1 | 0 | 1 | 0 | 1x1 blue PNG (`sea`) |
| 1 | 1 | 1 | 0 | 1x1 green PNG (`land`) |

Rows 1 and 2 share one stored image, so the same 69-byte blob must appear
twice (the dedup case). No `tile_id` column may survive into the output.
