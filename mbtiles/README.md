# MBTiles reader

Reads MBTiles tilesets (`.mbtiles`) in pure Power Query M. No GDAL, no tile
server, nothing to install.

MBTiles is the de facto container for map tiles: Mapbox tooling, tippecanoe,
QGIS and most basemap pipelines produce it. An MBTiles file is a SQLite
database with a specified schema, so this reader parses the file directly
and exposes the tile inventory and metadata as ordinary tables, which is
what an analytics tool actually wants from a tileset: coverage by zoom
level, tile counts, sizes, and the raw tile blobs.

## Usage

This reader depends on the SQLite core next to it: paste
[`../sqlite3/Sqlite3.Database.pq`](../sqlite3/Sqlite3.Database.pq) into a
blank query named `Sqlite3.Database`, then paste
[`Mbtiles.Document.pq`](Mbtiles.Document.pq) into a second blank query named
`Mbtiles.Document`. Then:

```m
let
    Source = File.Contents("C:\data\basemap.mbtiles"),
    Ts     = Mbtiles.Document(Source),
    Tiles  = Ts{[Name = "Tiles"]}[Data],
    Meta   = Ts{[Name = "Metadata"]}[Data]
in
    Tiles
```

`Tiles` has the stored `zoom_level`, `tile_column`, `tile_row` and
`tile_data` columns plus a computed `tile_row_xyz` column. MBTiles stores
rows in TMS order (row 0 at the bottom); `tile_row_xyz` is
`2^zoom - 1 - tile_row`, the XYZ/web-mercator row that web map URLs use.

Both storage layouts found in the wild are handled: the plain `tiles` table,
and the deduplicated tippecanoe layout where `tiles` is a SQL view over
`map` and `images` tables. In the second case the reader performs the join
itself in M, since there is no SQL engine here to evaluate the view.

### Options

Second argument, optional record, all keys optional:

| Key | Default | Effect |
|---|---|---|
| `DecompressTiles` | auto | Vector tiles (`format: pbf`) are stored gzipped, and `Binary.Decompress` handles gzip, so tiles carrying the gzip signature are decompressed. `true` forces decompression (error on non-gzip tiles); `false` leaves every blob untouched |
| `MaxRows` | all | Return at most N tile rows |
| `Strict` | `false` | Error on tolerated malformations (missing metadata table; a gzip-flagged tile that fails to decompress) |

The metadata key-value pairs are also attached to the navigation table as a
record (`Mbtiles.Metadata`), with `Mbtiles.Format` and `Mbtiles.Name` pulled
out for convenience.

## Supported

- MBTiles 1.x: `metadata` plus either a `tiles` table or the `map` +
  `images` deduplicated pair.
- Raster tilesets (png, jpg, webp): tile blobs exposed as binary, not
  decoded further.
- Vector tilesets (pbf): the gzip wrapper is removed by default, leaving raw
  Mapbox Vector Tile protobuf per tile.
- UTFGrid sidecar tables (`grids`, `grid_data`) when present, exposed raw.
- On the SQLite side (delegated to `Sqlite3.Database`): multi-level b-trees,
  overflow pages, UTF-8/16 encodings.

## Limitations

- **Tile content is not parsed.** PNGs stay PNGs and vector tiles stay
  protobuf; decoding Mapbox Vector Tile geometry is a different job from
  reading the container.
- **`tiles` views other than the map + images layout** produce a clear
  error naming the reason.
- **Memory.** The whole file is buffered; peak memory is a multiple of file
  size. City- and region-scale tilesets are fine; a planet extract is not.
- **Precision.** Integer cells decode exactly across the full signed 64-bit
  range (inherited from `Sqlite3.Database`). M numbers are IEEE doubles, so
  arithmetic on values beyond 2^53 afterwards can still coerce to double.

### Why two queries

Every other reader in this repository is a single self-contained paste. This
one (and the GeoPackage reader) is the deliberate exception: an MBTiles file
is a SQLite database, the SQLite b-tree parser is by far the largest piece
of code involved, and duplicating it here would mean fixing every b-tree bug
in several places. So the SQLite core lives once, in
[`sqlite3/Sqlite3.Database.pq`](../sqlite3/Sqlite3.Database.pq), and this
reader calls it by name. A packaged single-file build may come later.

## How it works

An MBTiles file is a SQLite 3 database, so the reader delegates all
SQLite-level work (b-trees, records, overflow pages) to `Sqlite3.Database`
and consumes its navigation table. On top of that it reads `metadata` into a
key-value table, materialises `tiles` either directly or by joining `map` to
`images` on `tile_id`, sniffs each `tile_data` blob for the gzip signature
(`1F 8B`) and inflates it with `Binary.Decompress` when found, and adds the
XYZ row conversion.

The format is defined by the MBTiles specification (Mapbox), versions 1.1
through 1.3.
