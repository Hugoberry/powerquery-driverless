# GeoPackage fixtures: what each file proves

All fixtures are synthetic, generated by `make_fixtures.py` in this directory
with the Python standard library (sqlite3 + struct); every GeoPackageBinary
blob is hand-built byte by byte, no GDAL involved. Regenerate with:

```
python3 -m venv venv && venv/bin/pip install shapely
venv/bin/python make_fixtures.py
venv/bin/python check_mirror.py
```

The geometry decode logic destined for `Gpkg.Database.pq` is mirrored in
`mirror.py` and validated by `check_mirror.py` on every geometry, both
against shapely (an independent WKB reader wrapping GEOS) and against the
hand-written literals below.

To test, paste `sqlite3/Sqlite3.Database.pq` as a query named
`Sqlite3.Database` (the reader depends on it), then load the fixture through
`Gpkg.Database`:

```m
Gpkg.Database(File.Contents("...\gpkg\test\basic.gpkg"))
```

## basic.gpkg

SQLite 3, page size 4096, application_id `GPKG`, user_version 10300.
`gpkg_contents` lists five entries; the navigation table must show exactly
these five (plus the `gpkg_*` tables only under
`[IncludeSystemTables = true]`), and must NOT show the `rtree_points_geom*`
spatial-index tables, `sqlite_master`, or any other internals.

### points (features, srs 4326, geometry column `geom`, declared POINT)

| fid | name | rating | geom (expected WKT) |
|---|---|---|---|
| 1 | alpha | 4.5 | `POINT (30 10)` |
| 2 | bravo | -0.25 | `POINT (-101.5 45.25)` |
| 3 | charlie-BE | 2 | `POINT (2.5 3.5)` |
| 4 | empty | null | `POINT EMPTY` |
| 5 | nogeom | 1.5 | null |

Proves: little-endian blob with no envelope (1), envelope indicator 1 (2),
big-endian header AND big-endian WKB (3), the empty-geometry flag with a
NaN-coordinate WKB point (4), and a NULL geometry cell (5). `fid` is an
`INTEGER PRIMARY KEY` rowid alias and must come back 1..5.

### shapes (features, srs 4326, geometry column `shape`, declared GEOMETRY)

| fid | kind | shape (expected WKT) |
|---|---|---|
| 1 | linestring | `LINESTRING (30 10, 10 30, 40 40)` |
| 2 | polygon | `POLYGON ((35 10, 45 45, 15 40, 10 20, 35 10), (20 30, 35 35, 30 20, 20 30))` |
| 3 | multipoint | `MULTIPOINT ((10 40), (40 30), (20 20), (30 10))` |
| 4 | multilinestring | `MULTILINESTRING ((10 10, 20 20, 10 40), (40 40, 30 30, 40 20, 30 10))` |
| 5 | multipolygon | `MULTIPOLYGON (((30 20, 45 40, 10 40, 30 20)), ((15 5, 40 10, 10 20, 5 10, 15 5)))` |
| 6 | collection | `GEOMETRYCOLLECTION (POINT (40 10), LINESTRING (10 10, 20 20, 10 40))` |
| 7 | emptymulti | `MULTIPOLYGON EMPTY` |
| 8 | linestring-z | `LINESTRING Z (1 2 3, 4 5 6)` |

Proves: every WKB base type, a polygon with an interior ring, nested
per-child WKB headers inside multi-geometries, a zero-count multi as EMPTY,
a Z linestring with envelope indicator 2, and a geometry column not named
`geom`.

### zm (features, srs 4326, geometry column `geom`)

| fid | tag | geom (expected WKT) |
|---|---|---|
| 1 | z | `POINT Z (1 2 3)` |
| 2 | m | `POINT M (1 2 4)` |
| 3 | zm | `POINT ZM (1 2 3 4)` |

Proves: Z (+1000), M (+2000) and ZM (+3000) type codes, coordinate counts 3,
3 and 4, and envelope indicator 4 (full XYZM envelope, row 3).

### tiles_demo (tiles, srs 4326)

Three rows: (zoom 0, col 0, row 0), (0, 1, 0), (1, 2, 1), each with the same
70-byte 1x1 PNG blob (starts `89 50 4E 47`). Proves: the `tiles` data type is
exposed raw with blobs intact, and `gpkg_tile_matrix` /
`gpkg_tile_matrix_set` stay hidden unless system tables are requested.

### Under [GeometryFormat = "WKB"]

`points` fid 1 must return 21 bytes starting `01 01 00 00 00` (little-endian
WKB point); fid 3 must return 21 bytes starting `00 00 00 00 01` (big-endian).
The GeoPackageBinary header and envelope are stripped in both cases.

### Under [IncludeSystemTables = true]

Additional entries appear, sorted: `gpkg_contents` (5 rows),
`gpkg_extensions` (1 row), `gpkg_geometry_columns` (3 rows),
`gpkg_spatial_ref_sys` (3 rows), `gpkg_tile_matrix` (2 rows),
`gpkg_tile_matrix_set` (1 row), each with `DataType` = `system`.
